# 第一部分：快速入门

## 1. 系统简介

### 1.1 什么是 Claude Code UI

Claude Code UI 是一个为专利从业人员设计的 **AI 辅助文档生成平台**。它基于 Claude AI 模型，通过强大的 **Extensions 扩展系统**，帮助您：

- **快速生成专利文档**：技术交底书、权利要求书、商业价值分析等
- **撰写技术方案**：系统架构、实施方案、可行性评估
- **产出学术论文**：工程论文、科学论文、经济论文
- **处理文档格式**：PDF、Word、PowerPoint 文档的智能处理

### 1.2 核心功能概览

| 功能模块 | 说明 |
|---------|------|
| **智能对话** | 与 Claude AI 进行自然语言交互 |
| **Extensions 系统** | 预置的专业文档生成扩展 |
| **项目管理** | 多会话管理 |
| **文件浏览器** | 可视化项目文件管理 |
| **内置终端** | 容器隔离的安全执行环境 |

# 第三部分：Agent 与文档生成

## 9. Agent 系统概述

### 什么是 Agent

**Agent（智能代理）** 是系统的"大脑编排器"，它不直接生成文档，而是：
- 理解您的复杂需求
- 规划执行步骤
- 协调调用各个 Skills
- 处理多阶段工作流
- 确保质量和完整性

### 9.1 Agent vs Skill 的区别

| 特性 | Agent（代理） | Skill（技能） |
|------|-------------|--------------|
| **角色** | 编排器、指挥官 | 执行者、工匠 |
| **能力** | 理解意图、规划流程 | 执行具体任务 |
| **调用方式** | 自动识别或用户明确调用 | 自动识别或用户明确调用 |
| **示例** | 使用`generate-docs-agent`帮我生成下交底书：基于物联网的智能绿植监测系统 | 使用`patent-tech-disclosure` 生成交底书|

> **提示**：如果不清楚有哪些功能，直接询问 AI："有哪些文档生成功能？"

AI 会告诉您可用的功能和使用方法。

### 9.2 可用的 Agents

系统为专利代理工作预置了以下 Agents：

| Agent | 中文名称 | 核心能力 | 适用场景 |
|-------|---------|---------|----------|
| **generate-docs-agent** | 文档生成编排器 | 理解需求、规划流程、调用 Skills | 生成任何专利/技术/论文文档 |
| **patent-challenger-agent** | 专利质疑者 | 尖锐质疑、迫使补充技术细节 | 技术交底书缺乏实质性内容 |
| **document-reviewer-agent** | 文档审核器 | 质量打分、结构化评审 | 对生成的文档进行质量评估 |

## 10. generate-docs-agent（文档生成编排器）

### 10.1 功能概述

**generate-docs-agent** 是您最常用的 Agent，它是所有文档生成任务的"总指挥"。

**核心能力**：
- ✅ 自动理解您的文档生成需求
- ✅ 规划完整的执行步骤
- ✅ 协调调用 23+ 专业 Skills
- ✅ 管理文档依赖关系
- ✅ 自动触发质疑和审核环节

### 10.2 使用方式

#### 方式一：自然语言描述（推荐）

直接描述您的需求，系统会自动调用 `generate-docs-agent`：

```
请生成技术交底书：基于物联网的智能绿植监测系统

请帮我写一份技术方案：AI驱动的智能垃圾分类系统

生成完整的专利申请文档：一种基于深度学习的医学影像诊断方法
```

#### 方式二：明确指定 Agent

```
使用 generate-docs-agent 生成技术交底书

调用 generate-docs-agent，帮我写技术方案
```

### 10.3 支持的文档类型

`generate-docs-agent` 可以生成以下所有文档类型：

**专利类**：
- 技术交底书 (`patent-tech-disclosure`)
- 权利要求书 (`patent-patent-writing`)
- 专利创新评估 (`patent-patent-innovation-assessment-report`)
- 商业价值分析 (`patent-business-analysis`)
- IP保护策略 (`patent-ip-strategy`)

**技术类**：
- 技术方案 (`technical-tech-solution`)
- 价值主张 (`technical-value-proposition`)
- 立论白皮书 (`technical-thesis-doc`)
- 可行性评估 (`technical-project-feasibility-assessment-report`)
- 伦理风险评估 (`technical-ethics-report`)
- 合规标准策略 (`technical-standard-doc`)

**论文类**：
- 工程论文 (`paper-engineering-paper`)
- 科学论文 (`paper-science-paper`)
- 经济论文 (`paper-economy-paper`)

### 10.6 为什么要多个 Agent 协作？

**单一 Agent 的问题**：
- ❌ 既要生成文档，又要检查质量 → 容易遗漏问题
- ❌ 自己检查自己 → 难以发现自身问题

**多 Agent 协作的优势**：
- ✅ **职责分离**：生成、质疑、审核各司其职
- ✅ **对抗性**：质疑者专门找问题，更客观
- ✅ **完整性**：每个环节都有质量保证

## 11. patent-challenger-agent（专利质疑者）

### 11.1 功能概述

**patent-challenger-agent** 是一个"找茬专家"，它的作用是：
- 站在专利审查员的角度
- 对技术交底书进行尖锐质疑
- 迫使补充实质性技术细节
- 确保技术描述充分、具体

### 11.2 为什么需要质疑者？

**常见问题**：
- ❌ "采用先进算法" → 具体是什么算法？
- ❌ "自动识别" → 怎么识别的？
- ❌ "智能处理" → 处理逻辑是什么？

**质疑者的作用**：通过对抗式提问，迫使补充这些细节。

### 11.3 质疑类型

质疑者会从以下5个方面进行质疑：

| 类型 | 权重 | 示例问题 |
|------|------|----------|
| **空洞术语** | 30% | "先进算法"具体是Transformer、RNN还是CNN？ |
| **缺少执行主体** | 25% | 谁执行这个操作？传感器、控制器还是云端？ |
| **数据流中断** | 20% | 数据从传感器传到控制器的格式是什么？ |
| **算法缺失** | 15% | 用什么算法进行特征提取？ |
| **参数缺失** | 10% | 采样频率是多少？阈值如何设定？ |

### 11.4 使用方式

通常**不需要手动调用**，`generate-docs-agent` 会自动在生成技术交底书后调用质疑者。

如果需要单独使用：

```
请对 @generated_docs/xxx/tech_disclosure.md 进行质疑

使用 patent-challenger-agent 质疑这份交底书
```

### 11.5 质疑流程

```
生成技术交底书
       ↓
调用 patent-challenger-agent（第1轮）
       ↓
  输出质疑点（P0/P1问题）
       ↓
根据质疑修改文档
       ↓
调用 patent-challenger-agent（第2轮）
       ↓
  如果通过 → 进入审核
  如果不通过 → 继续修改（最多3轮）
```

### 11.6 质疑输出示例

```
🔍 第1轮质疑

P0问题（必须修复）：
1. 第2.3节提到"智能识别算法"，未说明具体是什么算法
   建议：明确是CNN、RNN、Transformer还是其他？

2. 第3.1节描述"自动采集数据"，缺少执行主体
   建议：说明是传感器主动采集，还是控制器轮询采集？

P1问题（强烈建议修复）：
1. 第4.2节提到"参数优化"，未说明优化方法
   建议：说明使用梯度下降、遗传算法还是网格搜索？
```

### 11.7 质疑规则

**质疑者遵循的原则**：
- ❌ 不说"描述不够详细"
- ✅ 说"第2.3节提到'先进算法'，是Transformer、RNN还是CNN？"
- ❌ 不说"需要补充细节"
- ✅ 说"数据从传感器到控制器的传输格式是什么？JSON还是二进制？"

## 12. document-reviewer-agent（文档审核器）

### 12.1 功能概述

**document-reviewer-agent** 是质量把关专家，它的作用是：
- 对生成的文档进行结构化质量评审
- 按照审核标准打分
- 输出问题和改进建议
- 必要时触发重新生成

### 12.2 与质疑者的区别

| 维度 | 质疑者 (Challenger) | 审核器 (Reviewer) |
|-----|---------------------|------------------|
| **执行时机** | 生成后、审核前 | 质疑通过后 |
| **核心目的** | 迫使补充技术细节 | 质量打分和最终评审 |
| **工作方式** | 对抗式提问 | 结构化评分 |
| **输出形式** | 具体质疑点 | 分数、等级、问题列表 |
| **终止条件** | 无P0/P1问题 | 分数≥80分 |
| **适用文档** | 技术交底书 | 所有有审核标准的文档 |

### 12.3 审核标准

审核器会从以下维度进行评分：

| 维度 | 权重 | 满分标准 |
|------|------|----------|
| **完整性** | 30% | 所有必需章节齐全，内容完整 |
| **技术性** | 25% | 技术描述准确、逻辑清晰 |
| **可实施性** | 20% | 技术方案可实施、有具体方案 |
| **规范性** | 15% | 符合专利撰写规范 |
| **创新性** | 10% | 有明确创新点和有益效果 |

**总分计算**：各维度加权求和，满分100分

### 12.5 审核流程

```
质疑环节完成（或文档已生成）
       ↓
调用 document-reviewer-agent
       ↓
加载审核标准
       ↓
执行结构化评分
       ↓
输出审核报告
       ↓
  如果分数 ≥ 80 → 通过，返回结果
  如果分数 < 80 → 提供反馈，重新生成
       ↓
最多重新生成3次
```

### 12.6 使用方式

通常由 `generate-docs-agent` 自动调用。如需单独使用：

```
请审核 @generated_docs/xxx/tech_disclosure.md

使用 document-reviewer-agent 评估这份文档的质量
```

## 13. 常见问题 FAQ

### Q1: 使用 Agent 和直接调用 Skill 有什么区别？

**A**: Agent 的优势：
- **理解能力**：理解您的复杂需求
- **规划能力**：自动规划执行步骤
- **质量控制**：自动触发质疑和审核
- **一站式服务**：不需要您了解底层 Skills

直接调用 Skill 适合：
- 明确知道要做什么
- 只需要生成单一文档
- 不需要质量保证环节

### Q2: 可以跳过质疑或审核环节吗？

**A**: 可以，但不推荐。如果确实需要，可以明确说明：

```
生成技术交底书，不需要质疑环节

生成技术方案，直接输出，不需要审核
```

### Q3: 质疑环节最多进行几轮？

**A**: 最多3轮。3轮后即使仍有P0/P1问题，也会进入审核环节。

### Q4: 审核分数低于80会怎样？

**A**: 系统会：
1. 提供详细的审核反馈
2. 根据反馈重新生成文档
3. 最多重新生成3次
4. 3次后接受最后一次生成的版本

### Q5: 如何查看生成的文档？

**A**: 有两种方式：
1. 在聊天界面查看（AI 会显示内容摘要）
2. 在文件浏览器中打开完整文件（右上角 files 标签）

### Q6: 生成的文档可以直接使用吗？

**A**: 生成的文档是高质量草稿，建议：
1. 仔细审核技术内容的准确性
2. 补充具体的实验数据
3. 由专业专利代理人最终审核
4. 根据实际需求调整格式

### Q7: 可以在后续对话中继续使用之前生成的文档吗？

**A**: 是的。在同一会话中，Agent 会记住：
- 之前生成的文档
- 质疑和审核的结果
- 您的反馈和修改要求

### Q8: 如何引用已有文档生成新文档？

**A**: 使用文件引用：

```
请参考 @existing_doc.md 生成新的技术交底书

基于 @tech_proposal.md 生成完整的专利申请文档
```

### Q9: 生成的文档太简单怎么办？

**A**：

**原因**：输入信息不够详细

**解决**：
```
请重新生成，补充以下信息：
发明名称：基于物联网的智能健康监测水杯
技术背景：现有水杯功能单一，无法监测饮水量、水温、水质...
技术方案：集成温湿度传感器、水质检测模块、蓝牙通信模块...
实施方式：传感器采集 → 微控制器处理 → 蓝牙传输 → 手机APP显示
有益效果：帮助用户养成健康饮水习惯，预防水质问题...
```

### Q10: 质疑环节一直无法通过怎么办？

**A**：

**原因**：技术描述确实缺少实质性内容

**解决**：
- 提供更具体的技术方案
- 补充算法、参数、数据流等细节
- 参考质疑者的问题逐一回答

### Q11: 审核分数始终低于80怎么办？

**A**：

**原因**：文档结构或规范性问题

**解决**：
- 检查是否所有必需章节都齐全
- 确认技术描述逻辑清晰
- 补充实验数据或实施例

## 14. 总结

1. **提供足够的输入信息**：信息越详细，生成质量越高
2. **利用质疑环节**：认真对待质疑问题，补充具体细节
3. **关注审核反馈**：根据审核分数和问题进行改进
4. **保存好版本**：每次生成都会保存，方便对比和回溯
5. **人工审核把关**：AI 是辅助，最终需要专业审核
